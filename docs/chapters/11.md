# <a id="-refheading-toc1622-3043861001"></a>SUMMARY SECTION

## <a id="-refheading-toc37630-3701168388"></a>Introduction

The [SUMMARY](#-refheading-toc43949-784232322) section defines the variables to be written to the summary files that are used to generate line graphs of properties such as oil flow rate versus time, grid block pressure versus time, etc. Unlike the other sections, the [SUMMARY](#-refheading-toc43949-784232322) section has basically two types of keywords: keywords that request a variable to be written out to the summary file as described in section [11.2](#112data-requirements)[#11.2.Data Requirements|outline](#112data-requirements)[Data Requirements](#112data-requirements), and the conventional type of keyword described in section [11.3](#113keyword-definitions)[#11.3.Keyword Definitions|outline](#113keyword-definitions)[Keyword Definitions](#113keyword-definitions) that perform an action. There are literally hundreds of keywords associated with the former indicating the variety of data that can be written to the summary file. Whereas, for the latter there are less than twenty keywords.
Standard summary file output consists of three files as outlined in Table 11.1, with the raw requested data stored in [SUMMARY](#-refheading-toc43949-784232322) index and [SUMMARY](#-refheading-toc43949-784232322) data files, the [SUMMARY](#-refheading-toc43949-784232322) RSM file is an ASCII file that is generated from the other two files by the simulator at the end of the run. There is also the option to output an OPM Flow specific Enhanced [SUMMARY](#-refheading-toc43949-784232322) output file.
| File Type | Data Type | Description | OPM Flow Status |
| --- | --- | --- | --- |
| ESMRY | Dynamic Vector Data | This is an OPM FLOW specific Enhanced [SUMMARY](#-refheading-toc43949-784232322)  \(\*.ESMRY\) output file format that is optimized for fast loading of selected vectors by post\-processing applications. The \*.ESMRY file is re\-written for every time step and does not have a performance impact on the simulation. Load times are in the order of 30 times faster compared with fetching selected vectors using the \*.UNSMRY file format in post processing software. The file type is compatible with OPM ResInsight and the option is activated via the \-\-enable\-esmry=true command line option \(true is the default value\). Note that the [FMTOUT](#-refheading-toc46651-1640804870),  [MULTOUT](#-refheading-toc195175-1371377330), and [UNIFOUT](#-refheading-toc65809-1640804870) keywords have no impact on this file type. | OPM Flow Fully Supported. |
| RSM | Output File | The RSM file contains the variables requested to be written to the [SUMMARY](#-refheading-toc43949-784232322) file via the keywords described in the [11.2](#112data-requirements)[#11.2.Data Requirements\|outline](#112data-requirements)[Data Requirements](#112data-requirements), in a tabulated ASCII output format. The format of the file enables the data to be easily loaded into the LibreOffice Calc program for further processing, as each vector represents one column. The report is written at the end of the simulation run by parsing the [SUMMARY](#-refheading-toc43949-784232322) Index and [SUMMARY](#-refheading-toc43949-784232322) Data files. | Fully Supported |
| SUMMARY Index | Dynamic Vector Index | The [SUMMARY](#-refheading-toc43949-784232322) index file type specifies and defines the format and data type written to the [SUMMARY](#-refheading-toc43949-784232322) Data file. | Fully Supported |
| SUMMARY Data | Dynamic Vector Data | The [SUMMARY](#-refheading-toc43949-784232322) data file contains the variables requested to be written to the file via the keywords described in the [11.2](#112data-requirements)[#11.2.Data Requirements\|outline](#112data-requirements)[Data Requirements](#112data-requirements). The data consists of vectors that are used to generate line graphs of properties such as oil flow rate versus time, grid block pressure versus time, etc. The properties to be stored on the [SUMMARY](#-refheading-toc43949-784232322) file are written to the summary file at the end of each successful time step. The data can be used to compare actual production data with the simulation derived results in post processing software like OPM ResInsight. | Fully Supported |
| Notes: - If no summary data is requested then the files will not be created. - The [SUMMARY](#-refheading-toc43949-784232322) files can be loaded or written out in either unified or non\-unified formats as well as in binary or ASCII formats. |  |  |  |

Table 11.1: OPM Flow Summary Output File Summary

As mentioned in Table 11.1 all files can be written out in either ASCII or binary formats and in addition the [SUMMARY](#-refheading-toc43949-784232322) file can be loaded or written out in either unified or non\-unified formats.  The file type \(ASCII or binary\) and file structure format \(unified or non\-unified formats\) is set via a set of keywords in the [RUNSPEC](#-refheading-toc55591-1778172979) section, as described in Table 11.2 for easy of reference.

| Process | RUNSPEC Keyword | Description | Files |
| --- | --- | --- | --- |
| Input | [FMTIN](#-refheading-toc46649-1640804870) | The keyword defines the input files to be formatted as ASCII, i.e., text files, as opposed to binary files. | \*.FSMSPEC \*.FUNSMRY |
| [MULTIN](#-refheading-toc195177-1371377330) | The keyword defines the input files to be non\-unified multiple files, as opposed to unified files. | \*.SMSPEC \*.S0001 |  |
| [UNIFIN](#-refheading-toc46653-1640804870) | [UNIFIN](#-refheading-toc46653-1640804870) defines the input files to be unified files, as opposed to non\-unified multiple files. | \*.SMSPEC \*.UNSMRY |  |
| Output | [FMTOUT](#-refheading-toc46651-1640804870) | The keyword sets all output files to be formatted as ASCII, i.e., text files, as opposed to binary files. | \*.FSMSPEC \*.FUNSMRY |
| [MULTOUT](#-refheading-toc195175-1371377330) | [MULTOUT](#-refheading-toc195175-1371377330) defines the output files to be non\-unified multiple files, as opposed to unified files. | \*.SMSPEC \*.S0001 |  |
| [UNIFOUT](#-refheading-toc65809-1640804870) | [UNIFOUT](#-refheading-toc65809-1640804870) defines the output files to be unified files, as opposed to non\-unified multiple files. | \*.SMSPEC \*.UNSMRY |  |
| - A binary file is computer\-readable but not human\-readable. - For unified files if the run terminates unexpectedly, or there is insufficient disk space, then the last reported output is not stored.  The main advantage of unified files is that if a number of simulation cases reside in the same directory, the number of output files per case can be far fewer than using non\-unified multiple files.  There is no limit on the number of reporting steps that a unified file can store. |  |  |  |

Table 11.2: RUNSPEC Input and Output File Format Keywords

OPM Flow automatically generates the [SUMMARY](#-refheading-toc43949-784232322) file names based on the input file name and the output options selected via the keywords in the [RUNSPEC](#-refheading-toc55591-1778172979) section as summarized in in Table 11.2.  For example, starting OPM Flow using the following command from the terminal:

```eclipse
		flow CASENAME
```

will start the simulator and run the case specified by CASENAME.DATA. Here CASENAME is the “root” of the filename and DATA is the extension of the filename.  OPM Flow will generate the [SUMMARY](#-refheading-toc43949-784232322) output files based on the CASENAME with the extension based on the type \(ASCII or binary\) and file structure format \(unified or non\-unified formats\), as outlined in Table 11.3.

| File Type | Unformatted Extension | Description | Formatted Extension |
| --- | --- | --- | --- |
| RSM | \*.RSM | Output RSM file. | \*.RSM |
| SUMMARY Index | \*.SMSPEC | The [SUMMARY](#-refheading-toc43949-784232322) index file for both unified and non\-unified formats. | \*.FSMSPEC |
| SUMMARY Data Non\-Unified | \*.Snnnn | The [SUMMARY](#-refheading-toc43949-784232322) data files containing the variables requested to be written to the [SUMMARY](#-refheading-toc43949-784232322) files. For example, the [SUMMARY](#-refheading-toc43949-784232322) non\-unified binary files would be: CASENAME.S0001 CASENAME.S0002 CASENAME.S0003 etc. CASENAME.SMSPEC | \*.Annnn |
| SUMMARY Data Unified | \*.UNSMRY | The [SUMMARY](#-refheading-toc43949-784232322) data file containing the variables requested to be written to the [SUMMARY](#-refheading-toc43949-784232322) file. For example, the [SUMMARY](#-refheading-toc43949-784232322) data unified binary files: CASENAME.UNSMRY CASENAME.SMSPEC | \*.FUNSMRY |
| Notes: - The above file naming convention is for Linux type operating systems, as OPM Flow is currently only officially supported for Linux distributions. |  |  |  |

Table 11.3: OPM Flow File Naming Conventions

The default behavior is to write out the requested variables at each time step. As this can lead to large files \(or a large number of files for non\-unified output\), especially for full field simulation models, the [RPTONLY \- Activate the Report Time Steps Only Option for the SUMMARY File](#1024rptonly---activate-the-report-time-steps-only-option-for-the-summary-file) keyword allows one to write out the data only at a report time step instead.
The [SUMMARY](#-refheading-toc43949-784232322) section is terminated by the [SCHEDULE](#-refheading-toc43945-784232322) keyword.
